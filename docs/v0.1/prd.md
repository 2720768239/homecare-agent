## 1. 项目概述

### 1.1 项目名称

**HomeCare Agent**

中文说明名：家庭设备管家 Agent

后续正式项目文档、测试文档、UI 设计文档、研发文档、验收文档均统一使用 **HomeCare Agent** 作为项目名称。

---

### 1.2 产品形态

v0.1 只定义为一个**移动端应用**。

本阶段不同时设计 PC 端、Web 管理台或桌面端体验，避免产品方向发散。

移动端具体技术承载形式可在后续研发文档阶段确定，例如：

* 移动端 H5
* PWA
* 小程序
* React Native / Flutter App
* WebView 容器 App

需求阶段只明确：**用户主要在手机上使用 HomeCare Agent 完成拍照上传、自动建档、查看设备、询问说明书、故障求助和查看提醒。**

---

### 1.3 一句话定位

HomeCare Agent 是一个移动端家庭设备管理 Agent。

用户可以用手机拍照或上传订单截图、发票、说明书、保修卡、设备照片，通过自然语言让 Agent 自动创建设备档案；设备出现问题时，Agent 可以帮助用户查询保修状态、检索说明书、生成排查步骤和售后沟通文案。

---

### 1.4 产品核心价值

HomeCare Agent 不是普通文件夹，也不是简单的家电资料库，而是一个围绕“家庭设备生命周期”的移动端 Agent。

核心价值包括：

1. **拍照 / 上传后自动建档**
   用户用手机拍照或上传设备资料，Agent 自动识别信息并生成设备档案草稿。

2. **用户确认后入库**
   Agent 不直接写入关键数据，必须先展示草稿，由用户确认、修改或取消。

3. **说明书可问**
   用户不需要翻 PDF，可以直接问设备使用方法、清洁方式、错误代码和保养步骤。

4. **保修可查**
   系统根据购买日期和保修期计算保修状态，并展示即将过保设备。

5. **故障可处理**
   设备故障时，Agent 可以结合设备档案、保修状态和说明书内容，生成排查步骤、售后沟通文案和材料清单。

---

## 2. 背景与问题

### 2.1 用户现状

家庭设备通常会产生大量零散资料，包括订单截图、发票、保修卡、说明书 PDF、纸质说明书照片、设备照片、序列号、售后电话、维修记录和耗材更换记录。

这些资料经常分散在电商 App、手机相册、文件夹、微信聊天、邮箱、纸盒和抽屉中。

---

### 2.2 典型问题

1. 买完设备后没有及时建档。
2. 设备坏了不知道是否还在保修期内。
3. 说明书找不到或懒得看。
4. 保修快过期但用户不知道。
5. 故障时不知道如何描述问题。
6. 维修和保养历史没有记录。

---

## 3. 产品目标

### 3.1 v0.1 产品目标

v0.1 的目标是完成一个移动端可用、可演示、可继续扩展的 Agent MVP。

本版本重点完成：

1. 用户可以在手机端通过自然语言和文件上传发起建档任务。
2. 用户可以拍照上传订单截图、发票、说明书、保修卡和设备照片。
3. Agent 可以识别用户意图并解析上传资料。
4. Agent 可以生成设备档案草稿。
5. 用户可以在移动端确认、修改或取消设备草稿。
6. 用户确认后，系统创建设备档案。
7. 系统可以绑定附件、计算保修状态、创建应用内保修提醒。
8. 用户可以在设备详情页查看设备资料。
9. 用户可以基于说明书向 Agent 提问。
10. 用户可以发起故障售后辅助流程。
11. 用户可以查看 Agent 的执行过程和历史记录。

---

### 3.2 v0.1 主线优先级

| 优先级 | 能力 | 说明 |
| --- | --- | --- |
| A | 自动建档 | 主线能力，必须完整打磨 |
| B | 故障售后 | 展示亮点，做出完整 Demo 链路 |
| C | 说明书问答 | 基础能力，为 A 和 B 提供支撑 |

## 5. 用户与使用场景

### 5.1 目标用户

v0.1 面向一个固定家庭空间使用。当前只支持两个预置家庭成员账号，两个账号共同查看和管理同一套家庭设备数据。

目标用户包括家庭设备较多的用户、夫妻 / 家庭成员共同管理家电资料的家庭、租房用户、新装修家庭、喜欢整理家庭资料的用户、经常购买数码产品和家电的用户、需要管理保修和说明书的用户。

---

### 5.2 用户角色

| 角色 | 说明 |
| --- | --- |
| 家庭成员 A | 使用预置账号登录，可查看、创建、编辑、管理同一个家庭空间内的全部设备资料 |
| 家庭成员 B | 使用预置账号登录，可查看、创建、编辑、管理同一个家庭空间内的全部设备资料 |

---

### 5.3 核心用户故事

1. **登录进入家庭空间**：作为家庭成员，我希望通过账号名和密码登录 HomeCare Agent，进入同一个家庭设备空间。
2. **共同查看和管理设备**：作为家庭成员，我希望另一个家庭成员创建或修改的设备资料我也可以看到和管理。
3. **拍照自动建档**：作为用户，我希望用手机拍照上传订单截图、发票、说明书或保修卡后，HomeCare Agent 能自动识别设备信息并生成设备档案草稿。
2. **确认后入库**：作为用户，我希望 HomeCare Agent 在真正创建设备前先让我确认，避免错误信息进入设备库。
3. **查看设备资料**：作为用户，我希望在手机上快速查看设备的购买日期、保修状态、说明书、保修卡、订单截图和维修记录。
4. **询问说明书**：作为用户，我希望直接问“这个设备怎么清洗”“错误代码是什么意思”，HomeCare Agent 能基于我上传的说明书回答。
5. **故障售后辅助**：作为用户，当设备坏了时，我希望 HomeCare Agent 帮我判断是否还在保修期、生成排查步骤、整理联系售后的话术和材料清单。
6. **查看执行过程**：作为用户，我希望看到 HomeCare Agent 当前执行到了哪一步，特别是在需要我确认时能清楚知道它准备做什么。

---

## 6. 移动端页面结构

v0.1 只设计移动端页面。

本版本采用 **Agent-first 单主界面**，不使用传统底部五 Tab 导航。用户打开应用后首先进入 HomeCare Agent 主界面，像使用 ChatGPT 一样直接输入任务、上传资料、查看 Agent 状态和结构化结果。

页面包括：

1. 登录页 / 登录错误态 / 鉴权检查态
2. Agent 主界面
3. 左侧抽屉菜单
4. 设备列表页
5. 设备详情页
6. 提醒页
7. Agent 执行记录页
8. 设置页
9. 附件预览页 / 底部弹窗

---

### 6.1 导航结构

移动端不使用底部导航。

推荐结构：

```text
顶部栏：
[菜单] HomeCare Agent                         [新任务]

主体：
Agent 对话流 / 任务输入 / 文件上传 / 结构化结果卡

底部：
[ + ] 问 HomeCare Agent...                    [发送]
```

左侧抽屉菜单结构：

```text
HomeCare Agent

新任务

主要
- 设备库
- 提醒
- Agent 执行记录

管理
- 设置
- 关于 HomeCare Agent
```

说明：

| 入口 | 作用 |
| --- | --- |
| Agent 主界面 | 发起自动建档、说明书问答、保修查询、故障售后，是默认首屏 |
| 新任务 | 清空当前输入状态，开启新的 Agent 会话或任务 |
| 设备库 | 查看、搜索和筛选设备档案 |
| 提醒 | 查看保修、保养、耗材更换等应用内提醒 |
| Agent 执行记录 | 查看历史任务、执行路径、确认记录和错误信息 |
| 设置 | 默认保修提醒、设备分类、数据导出、关于信息；数据导出不作为左侧抽屉独立入口 |

设计原则：

1. 高频任务不通过导航查找，而是在 Agent 主界面直接输入或上传。
2. 设备库、提醒、执行记录、设置、关于等低频管理功能统一收进左侧抽屉。
3. 数据导出不作为左侧抽屉独立入口，统一收纳在设置页内，避免抽屉入口过多。
4. 不保留「首页｜设备｜Agent｜提醒｜更多」底部导航，避免削弱 Agent 主界面的沉浸感。
5. 页面跳转后保留返回主界面的路径，避免用户在深层资料页迷失。

## 7. 页面需求

## 7.0 登录与鉴权页面

### 页面目标

v0.1 增加轻量登录能力，用于进入固定家庭空间。登录不作为复杂账号体系，仅用于区分当前操作者、保护家庭设备资料入口，并支持两个家庭成员共同管理同一套设备数据。

### 登录方式

```text
账号名 + 密码
```

v0.1 不提供账号注册、忘记密码、邮箱验证、短信验证码、微信登录、Google 登录或家庭成员邀请。

### 预置账号

| 账号名 | 密码 | 身份 | 家庭空间 | 权限 |
| --- | --- | --- | --- | --- |
| home_a | home123456 | 家庭成员 A | 我的家 / household_default | 查看、创建、编辑、管理全部设备 |
| home_b | home123456 | 家庭成员 B | 我的家 / household_default | 查看、创建、编辑、管理全部设备 |

两个账号共享同一个家庭空间，不做数据隔离，不做角色差异。

### 登录流程

```text
打开应用
→ Auth Checking
→ 未登录：进入登录页
→ 输入账号名和密码
→ 校验预置账号
→ 登录成功：进入 Agent 主界面
→ 登录失败：展示统一错误提示
```

### 页面状态

1. `00A Login`：默认登录页。
2. `00A-1 Login Error`：账号名或密码错误。
3. `00A-2 Auth Checking`：启动时检查本地登录态。

### 登录成功后

登录成功后进入 Agent 主界面。刷新页面时应保持登录态；点击设置页「退出登录」后清除本地登录态并回到登录页。

## 7.1 Agent 主界面

### 页面目标

Agent 主界面是 v0.1 的默认首屏和核心页面。

它让用户打开应用后可以立即完成自然语言任务输入、拍照 / 相册 / 文件上传、自动建档任务发起、说明书问答、保修查询、故障售后、Agent 执行状态查看、设备草稿确认和结构化结果查看。

---

### 首屏结构

```text
[菜单] HomeCare Agent                         [新任务]

今天要处理哪台设备？

你可以拍照上传订单、发票、说明书或保修卡，
也可以直接问我设备怎么用、是否还在保修、坏了怎么办。

快捷建议：
[拍照建档] [问说明书]
[查保修]   [故障售后]

最近任务 / 需要关注 / 最近设备

[ + ] 问 HomeCare Agent...                    [发送]
```

---

### 页面内容

1. 顶部栏：菜单按钮、标题 HomeCare Agent、新任务按钮。
2. 首屏引导文案。
3. 快捷建议按钮：拍照建档、问说明书、查保修、故障售后。
4. Agent 对话流。
5. 文件上传入口。
6. Agent 状态提示。
7. Agent 结构化结果卡片。
8. 执行过程折叠入口。
9. 轻量摘要卡片：即将过保、最近设备、最近任务。

---

### 空状态

当没有设备时展示：

```text
今天要处理哪台设备？

你可以拍照上传订单、发票、说明书或保修卡，
也可以直接问我设备怎么用、是否还在保修、坏了怎么办。
```

操作：

* 在输入框直接描述任务。
* 点击「+」选择拍照、相册或文件。
* 点击「拍照建档」快捷建议。
* 点击「手动添加设备」入口。

## 7.2 Agent 任务流

### 页面目标

Agent 任务流承载一次具体任务的输入、执行、确认和结果展示。

移动端采用单栏任务流，整体表现为 **对话流 + 结构化结果卡**。

---

### 页面内容

1. 对话消息。
2. 快捷任务按钮：拍照建档、问说明书、查保修、故障售后。
3. 任务输入框。
4. 文件上传入口。
5. Agent 状态提示。
6. Agent 结果卡片。
7. 执行过程折叠面板。

---

### 输入区交互

底部输入区固定在页面底部，但不能遮挡结果卡片的主要操作。

输入区结构：

```text
[ + ] 问 HomeCare Agent...                    [发送]
```

点击「+」后展示底部弹窗：

1. 拍照
2. 从相册选择
3. 上传文件
4. 手动添加设备

---

### 页面状态

| 状态 | 说明 |
| --- | --- |
| idle | 未开始 |
| input_ready | 用户已输入任务或上传文件 |
| uploading | 文件上传中 |
| running | Agent 执行中 |
| needs_confirmation | 等待用户确认 |
| applying_action | 正在执行确认后的操作 |
| completed | 任务完成 |
| failed | 任务失败 |

---

### 执行过程展示

移动端默认只展示当前关键状态，例如：

```text
正在解析文件...
正在生成设备草稿...
等待你确认设备草稿...
设备已创建
```

完整执行过程默认折叠，用户点击后展开底部弹窗。

## 7.4 设备详情页

### 页面目标

设备详情页是单个设备的完整档案。它需要回答：这是什么设备、什么时候买的、还在保修吗、说明书在哪里、有哪些附件、能不能问说明书、坏了怎么办、有没有维修或保养记录。

---

### 第一屏结构

设备详情页第一屏采用 **设备档案优先**：

```text
小米净水器 H1000G
厨房设备 · 小米 · H1000G

保修中 · 还有 320 天
说明书 1 · 附件 3 · 记录 2

[问说明书] [故障售后]
```

---

### 页面内容

1. 设备头部：设备名称、设备图片或设备图标、品牌 / 型号 / 分类、保修状态、说明书 / 附件 / 记录数量。
2. 第一屏快捷操作：问说明书、故障售后。
3. 折叠分区：购买与保修、附件、提醒、维修 / 故障 / 保养记录、基础信息。
4. 更多操作：编辑设备、上传附件、添加提醒、添加记录、删除设备。

---

## 10. 核心流程需求

## 10.1 自动建档流程

### 入口

自动建档可以从以下入口进入：

1. Agent 主界面快捷建议「拍照建档」。
2. 底部输入框「+」菜单中的拍照 / 相册 / 上传文件。
3. 设备库「添加设备」。
4. 设备空状态「返回 Agent 主界面上传资料建档」。

---

### 用户操作

用户可以手机拍照上传、从相册选择图片、从文件中选择 PDF、输入自然语言任务、直接输入设备信息、同时上传文件和输入文字。

支持文件类型：jpg / png / webp / pdf。

支持附件类型：订单截图、发票、说明书、保修卡、设备照片、维修单、其他。

---

### Agent 处理

HomeCare Agent 需要完成：识别任务是否为 create_device、解析上传文件内容、识别文件类型、提取设备信息、生成设备草稿、标记缺失字段、生成建议提醒、等待用户确认。

---

### 用户确认

HomeCare Agent 生成设备草稿后，必须等待用户确认。

用户可以选择：确认创建、修改信息、取消。

任何创建设备、绑定附件、创建提醒、保存记录等写入行为，都必须经过用户确认。

---

### 确认后动作

用户确认后，系统执行创建设备档案、绑定相关附件、计算保修截止日期、创建过保前提醒、如包含说明书则解析并索引说明书、生成完成结果、记录 Agent 执行过程、引导用户查看设备详情页。

---

## 10.2 说明书问答流程

### 入口

1. Agent 主界面。
2. 设备详情页「问说明书」。
3. 主界面快捷建议「问说明书」。

优先推荐从设备详情页发起，因为设备上下文明确。

---

### Agent 处理

HomeCare Agent 需要判断意图为 manual_qa、判断是否已明确设备、未明确设备时查找候选设备并让用户选择、检查设备是否有说明书、检索说明书知识块、基于检索结果生成回答、返回来源说明。

无说明书时提示：

```text
这台设备还没有上传说明书。
上传说明书后，我可以帮你回答使用方法、故障代码和保养步骤。
```

未检索到答案时提示：

```text
我没有在这台设备的说明书中找到明确答案。
你可以换个问法，或上传更完整的说明书。
```

HomeCare Agent 不得编造说明书内容。

---

## 10.3 故障售后流程

### 入口

1. Agent 主界面快捷建议「故障售后」。
2. 设备详情页「故障售后」。
3. 主界面自然语言输入。

优先推荐从设备详情页发起，因为设备上下文明确。

---

### Agent 处理

HomeCare Agent 需要判断意图为 troubleshooting、查找相关设备、多候选设备时让用户选择、查询保修状态、检索说明书、生成安全提醒、生成排查步骤、生成售后沟通文案、列出需要准备的材料、询问是否保存为故障记录。

---

### 输出结构

故障售后结果需要包括设备信息、保修状态、安全提醒、排查步骤、售后沟通文案、材料清单、保存故障记录操作。

---

### 安全边界

涉及强电、燃气、水路拆装、高温设备、电池膨胀、明火、漏电、需要拆机维修等情况时，HomeCare Agent 必须保守提示：

```text
这个问题可能涉及安全风险，建议先停止使用设备，并联系官方售后或专业维修人员。
```

HomeCare Agent 不得指导用户执行高风险拆机、带电维修、燃气维修等操作。

---

## 11. 移动端交互规则

### 11.1 全局交互

1. 所有页面采用单栏移动端布局。
2. 高频操作优先在 Agent 主界面直接输入或上传，低频管理入口通过左侧抽屉进入。
3. 表单编辑优先使用底部弹窗。
4. 复杂内容使用卡片和折叠面板。
5. Agent 执行过程默认折叠，只展示当前关键状态。
6. 所有写入操作必须使用确认卡片。
7. 上传入口必须支持拍照、相册和文件选择。
8. 首屏优先展示当前任务和下一步操作，不堆叠复杂信息。
9. 底部输入区不能遮挡确认、保存、上传等关键操作。
10. 左侧抽屉菜单必须可通过点击遮罩、返回手势或菜单项关闭。

---

### 11.2 设备草稿确认卡片

设备草稿确认采用 **档案卡片 + 展开字段**。

默认展示设备名称、品牌、型号、分类、保修状态、保修截止日期、已识别附件、建议提醒、缺失字段。

点击「查看全部字段」后展示设备名称、品牌、型号、分类、购买日期、购买价格、购买渠道、保修期、保修截止日期、序列号、来源文件、建议提醒、识别置信度。

操作按钮：确认创建、修改信息、查看全部字段、取消。

---

### 11.3 修改信息底部弹窗

用户点击「修改信息」后，展示底部弹窗。

字段包括设备名称、品牌、型号、分类、购买日期、购买价格、购买渠道、保修期、序列号、备注。

保存修改后返回设备草稿确认卡片。

---

### 11.4 执行过程底部弹窗

用户点击「查看执行过程」后，展示当前 Agent 执行路径。

节点状态包括已完成、执行中、等待确认、失败、跳过。

---

### 11.5 故障售后结果卡片

故障售后结果采用 **报告型 + 行动清单**。

结构包括：

1. 结果标题。
2. 设备信息。
3. 保修状态。
4. 安全提醒。
5. 先做这几件事。
6. 售后沟通文案。
7. 需要准备的材料。
8. 复制售后文案。
9. 保存故障记录。
10. 查看执行过程。

## 13. 权限与安全边界

### 13.0 登录与访问边界

未登录用户不得访问 Agent 主界面、设备库、设备详情、提醒、执行记录、设置页和附件预览。

登录成功后，`home_a` 和 `home_b` 都进入同一个家庭空间 `household_default`，两者权限一致，都可以查看、创建、编辑和管理家庭设备资料。v0.1 不做数据隔离、不做角色权限、不做多家庭切换。

系统可以记录当前操作者，例如 `created_by_user_id`、`updated_by_user_id`、`confirmed_by_user_id`，但该字段只用于审计和展示，不用于权限差异。

---

### 13.1 HomeCare Agent 可以自动执行

识别意图、解析文件、提取设备信息、生成设备草稿、计算保修期、搜索设备、搜索说明书、生成回答、生成售后文案、生成排查建议。

---

### 13.2 必须用户确认后才能执行

创建设备、修改设备、删除设备、绑定附件到设备、删除附件、创建提醒、保存故障记录、修改维修记录。

---

### 13.3 v0.1 禁止执行

自动联系售后、自动发送邮件、自动发送短信、自动购买耗材、自动删除大量数据、指导高风险维修、指导燃气维修、指导带电维修、指导复杂拆机维修。

---

## 14. 保修状态规则

| 条件 | 状态 |
| --- | --- |
| 缺少购买日期或保修期 | 未知 |
| 当前日期 > 保修截止日期 | 已过保 |
| 距离保修截止日期 <= 30 天 | 即将过保 |
| 其他情况 | 保修中 |

展示文案示例：保修中 · 还有 357 天；即将过保 · 12 天后过保；已过保 · 2025-12-01 到期；未知 · 缺少购买日期或保修期。

## 15. 文件处理规则

支持格式：jpg / png / webp / pdf。

文件状态：待上传、上传中、已上传、解析中、已解析、解析失败。

当文件解析失败时，系统提示：

```text
这个文件暂时无法解析。
你可以重新上传更清晰的图片、手动填写设备信息，或仅将文件作为附件保存。
```

可选操作：重新上传、手动填写、仅保存附件。

## 16. 非功能需求

### 16.1 移动端易用性

1. 手机端必须可以方便完成拍照上传。
2. 自动建档流程不能要求用户填写过多字段。
3. Agent 输出必须结构化展示，不能只返回大段文字。
4. 写入操作必须有确认卡片。
5. 错误状态必须提供下一步操作。
6. 关键按钮必须适合拇指点击。
7. 页面内容不能依赖横向滚动。
8. 输入框、文件上传、确认卡片是核心交互优先级。
9. 主界面不展示传统底部五 Tab 导航。

---

### 16.2 可解释性

1. Agent 执行过程需要可视化。
2. 用户应能看到当前执行到哪一步。
3. 用户应能知道 HomeCare Agent 准备创建什么、修改什么、保存什么。
4. 执行记录页需要展示任务路径。
5. 移动端执行过程默认收起，避免干扰主任务。

---

### 16.3 数据安全

1. 用户数据只能展示给当前用户。
2. 删除设备、删除附件等高风险操作必须确认。
3. HomeCare Agent 不得自动执行高风险操作。
4. 故障排查内容不得鼓励危险维修行为。

---

### 16.4 可扩展性

v0.1 应为后续能力预留空间：多家庭 / 成员邀请 / 权限管理、系统推送提醒、官方说明书搜索、电商订单导入、耗材管理、维修记录分析、多 Agent 协作、接入外部入口例如 Hermes Agent、后续如有需要再增加 PC/Web 管理端。

---

## 17. MVP 验收标准

### 17.1 自动建档验收

用户在移动端上传订单截图和说明书，并输入：

```text
帮我把这个设备建档
```

系统应完成识别 create_device、解析文件、提取设备信息、生成设备草稿、暂停等待用户确认、用户确认后创建设备、绑定附件、计算保修截止日期、创建保修提醒、索引说明书、设备出现在设备列表、设备详情页可查看、执行记录可查看完整路径。

---

### 17.2 说明书问答验收

用户在设备详情页提问：

```text
这个设备滤芯怎么清洗？
```

系统应找到当前设备、检索该设备说明书、基于说明书生成回答、显示答案来源。没有说明书时明确提示，不得编造。

---

### 17.3 故障售后验收

用户输入：

```text
我的净水器漏水了，帮我看看怎么办。
```

系统应识别 troubleshooting、找到相关设备、判断保修状态、检索说明书、生成安全提醒、生成排查步骤、生成售后沟通文案、列出材料清单、询问是否保存故障记录、用户确认后保存故障记录。

---

### 17.4 移动端体验验收

系统应满足：

1. 移动端可以完成上传资料建档。
2. 移动端可以查看设备列表。
3. 移动端可以查看设备详情。
4. 移动端可以发起说明书问答。
5. 移动端可以发起故障售后。
6. 移动端左侧抽屉菜单可正常打开、关闭和跳转。
7. 移动端不展示传统底部五 Tab 导航。

## 20. 当前需求结论

v0.1 需求最终收敛为：

> HomeCare Agent 是一个以“自动建档”为主线的移动端 Agent-first 应用。用户可以像使用 ChatGPT 一样，通过手机拍照、上传设备资料和自然语言输入，让 Agent 自动生成设备档案草稿，经确认后入库，并创建保修提醒和说明书知识库。设备建档后，用户可以在手机端查询保修、询问说明书，并在设备故障时获得排查步骤、售后沟通文案和材料清单。

v0.1 的核心闭环是：

```text
打开 Agent 主界面
↓
手机拍照 / 上传资料 / 输入描述
↓
HomeCare Agent 生成设备草稿
↓
用户确认
↓
创建设备
↓
绑定附件
↓
创建保修提醒
↓
索引说明书
↓
进入设备详情
↓
问说明书 / 故障售后
```
