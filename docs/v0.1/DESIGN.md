# HomeCare Agent 设计规则（design.md）

> 本文档用于约束 HomeCare Agent 后续所有移动端 UI 设计稿。  
> 目标：避免重复出现近期修正中暴露的问题，包括图标占位、文字换行异常、按钮溢出、底部拥挤、卡片内容硬塞、链接漂浮、警示卡过重等。

---

## 1. 产品与视觉基准

### 1.1 产品形态

- v0.1 仅设计 **移动端单一产品**。
- 不设计 PC/Web 双端结构。
- 设计核心是 **Agent-first**，主界面应像 ChatGPT App：以对话、任务结果、结构化卡片为主。
- 不使用底部导航栏。
- 设备库、提醒、执行记录、设置、关于等入口收纳到左侧菜单 / drawer；数据导出收纳在设置页，不作为抽屉独立入口。
- v0.1 增加轻量登录：账号名 + 密码，两个预置家庭成员账号，共享同一个家庭空间。

### 1.2 视觉风格

- 浅色模式。
- 黑白灰为主，少量语义色点缀。
- 整体气质：克制、干净、高级、移动端原生感。
- 不做复杂装饰，不做重色块，不堆过多按钮。

推荐颜色：

```text
Background: #FFFFFF
Surface: #F8F8F7
Muted: #F4F4F2
Text Primary: #111111
Text Secondary: #6B7280
Text Tertiary: #9CA3AF
Border: #E5E7EB
Black Action: #111111
Warning Background: #FFF7ED
Warning Text: #92400E
Danger Background: #FEF2F2
Danger Border: #FECACA
Danger Text: #991B1B
Success Text: #166534
```

## 2. 画板与固定组件规则

### 2.0 登录页画板规则

v0.1 需要补齐以下登录相关画板：

```text
00A Login
00A-1 Login Error
00A-2 Auth Checking
```

登录页规则：

- 登录方式为账号名 + 密码。
- 不出现注册入口、忘记密码、短信验证码、邮箱验证码、微信登录、Google 登录。
- 登录页需要说明当前是 v0.1 本地 Demo / 预置家庭账号。
- 错误态统一提示「账号名或密码错误」。
- Auth Checking 用于启动时检查本地登录态。
- 登录成功后进入 Agent Home。

预置账号展示可放在开发 / Demo 提示中，不作为正式产品营销内容：

```text
home_a / home123456
home_b / home123456
```

两个账号代表同一家庭的两个成员，看到同一个家庭空间「我的家」。

### 2.1 画板尺寸

所有移动端主画板默认使用：

```text
width: 390px
height: 844px
```

### 2.2 页面安全边距

```text
左右主边距：28px
主内容宽度：334px
卡片内边距：16px
卡片圆角：16–18px
```

### 2.3 顶部状态栏必须复用已验收版本

不得用字符或临时符号模拟状态栏。

标准结构：

```text
时间：x 32, y 22
Dynamic Island: x 154, y 14, w 82, h 26
信号：3 个竖条
Wi-Fi：SVG icon
电池：描边电池 + 填充
```

禁止：

- 使用字符 icon。
- Wi-Fi 出现散乱斜线。
- 状态栏图标不对齐。
- 出现蓝色选中框、蓝色描边残留。

### 2.4 顶部导航必须复用已验收版本

标准结构：

```text
左侧：三横菜单
中间：HomeCare Agent
右侧：新任务
```

位置参考：

```text
菜单线：x 28, y 74 / 83 / 92
标题：x 0, y 72, w 390, center
新任务：x 318, y 75, auto width
```

规则：

- 「新任务」必须单行展示。
- 不允许“新任 / 务”换行。
- 标题必须视觉居中，不被右侧文字挤偏。

### 2.5 底部 composer 必须复用已验收版本

所有 Agent 页面底部输入框必须统一：

```text
composer-bg: x 18, y 764, w 354, h 56, radius 28
plus-bg: x 28, y 774, w 36, h 36
plus SVG: x 34, y 780, size 24
input: x 76, y 774, w 240, h 36
placeholder: 问 HomeCare Agent...
send-bg: x 326, y 774, w 36, h 36
send SVG: x 332, y 780, size 24
```

规则：

- `+` 必须是标准加号，不允许显示成 `−`。
- 发送按钮必须是居中的白色上箭头，不允许像对勾。
- composer 不得遮挡卡片、按钮、文字。
- composer 上方至少保留 12px 视觉间距。

## 3. 信息架构规则

### 3.1 每一屏只能有一个主要任务

设计前必须先判断当前屏幕的主任务。

示例：

```text
01 首页：发起任务 / 提问 / 上传
02 建档确认：确认创建设备档案
03 故障结果：先处理故障风险，再保存记录
04 设备库：查看需要关注和最近设备
05 设备详情：查看设备档案与关键状态
```

页面内容必须服务主任务。非主任务内容应折叠、下沉或移入底部弹窗。

### 3.2 默认态必须轻，不要塞满全部信息

默认页只展示：

- 摘要。
- 最关键字段。
- 当前状态。
- 一个主操作。
- 必要的次级入口。

以下内容不得全部塞入默认页：

- 完整字段表。
- 长售后文案。
- 完整材料清单。
- 执行过程详情。
- 全量历史记录。

应使用：

```text
查看全部字段
查看执行过程
查看 / 复制售后文案
底部弹窗 Sheet
详情页
```

## 4. 对话区规则

### 4.1 用户消息气泡

用户消息靠右，浅灰背景。

推荐：

```text
background: #F4F4F2
radius: 20–22px
font-size: 14px
max-width: 230–268px
padding: 14–18px horizontal
```

规则：

- 不要让气泡过宽导致文字像横向铺满。
- 不要让中文在奇怪位置断行。
- 如果文本较长，应主动拆成自然两行。
- 气泡不要贴近状态行或卡片，至少保留 12–16px 间距。

### 4.2 当前设备上下文

设备上下文使用 pill，不使用普通文本。

推荐结构：

```text
[当前设备：小米净水器 H1000G  >]
```

样式：

```text
height: 30px
background: #F4F4F2
radius: 15px
font-size: 13px
text color: #6B7280
```

## 5. 状态行规则

状态行应轻，不要漂浮。

推荐结构：

```text
[✓] 已完成分析              [执行过程 >]
```

规则：

- 左侧是任务状态。
- 右侧入口应做成轻量 pill 或明确可点击区域。
- 不要让「查看执行过程」像漂在右边的普通文字。
- 不要加无意义进度线。
- 不使用字符 icon，统一使用 SVG。

## 6. 卡片规则

### 6.1 卡片必须完整包住内容

所有卡片必须满足：

```text
内容不得溢出卡片
按钮不得掉出卡片
卡片底部至少保留 12px 内边距
卡片与 composer 至少保留 12px 间距
```

禁止：

- 按钮贴着卡片底部。
- 文本或 chip 超出卡片边界。
- 卡片里强塞过多 section。
- 为了塞内容过度压缩字号、行高、间距。

### 6.2 卡片内 section 要有明确层级

推荐顺序：

```text
标题
状态 / 标签
核心内容
次级入口
操作按钮
```

不要出现：

```text
标题
长文案
按钮
材料清单
输入框
```

### 6.3 内容过多时必须拆分

当一张卡片内超过 5 个 section，必须拆分：

- 默认页只保留核心 section。
- 详细字段进入 02B / 03B 等扩展态。
- 长内容进入底部弹窗。


### 6.4 卡片内部内边距与字段行规则

卡片不仅外层要有内边距，卡片内部的二级容器、状态条、字段行、摘要文案也必须保留可见内边距。

强制规则：

```text
卡片外层左右内边距：至少 16px
卡片内二级容器左右内边距：至少 16px
字段行右侧值距离卡片右边界：至少 16px，推荐 24px
摘要文案距离卡片左右边界：至少 16px
状态条内部文字距离状态条左右边界：至少 16px
```

字段行推荐结构：

```text
左侧 label：x = card.x + 32
右侧 value：右对齐，right <= card.x + card.width - 32
label/value 行高：18px 以上
行间距：6–10px
```

禁止：

- 右侧字段值贴近卡片边缘。
- 摘要文案从卡片边线开始。
- 状态条里的左右两段文字互相挤压。
- 为了节省高度把字段行压到没有呼吸感。

验收标准：

```text
1. 字段值不贴右边界
2. 摘要文字左右有明确内边距
3. 二级状态条内部文字不贴边
4. label/value 在同一视觉网格上
5. 中文不会因为空间不足产生异常换行
```

### 6.5 详情卡片内边距专项规则

详情页、设备档案页、保修信息卡、附件摘要卡等信息密集型卡片，必须额外检查内部二级内容的内边距。

强制规则：

```text
二级状态条：x = card.x + 16，width = card.width - 32
状态条内部文字：距离状态条左右边界至少 16px
字段行右侧值：right <= card.x + card.width - 32
字段行最后一行：距离卡片底部至少 12px，推荐 16px
摘要文案：x >= card.x + 32，right <= card.x + card.width - 32
摘要文案底部：距离卡片底部至少 12px，推荐 16px
```

当状态条内同时出现状态与日期，例如：

```text
保修中 · 还有 320 天    至 2028-06-20
```

如果横向空间不足，必须改成上下两行：

```text
保修中 · 还有 320 天
保修至 2028-06-20
```

禁止：

- 为了横排而让日期、单位、金额发生异常换行。
- 字段值贴到卡片右边界。
- 摘要文案贴着卡片左边界或底边。
- 最后一行文字正好压在卡片底部，没有下内边距。

设计验收时必须检查：

```text
1. 卡片内部每一层都有可见内边距
2. 状态条文字不贴边、不挤压、不异常换行
3. 字段值距离卡片右边界至少 32px
4. 摘要文案距离左右边界和底部都有明确留白
5. 卡片最后一行内容下方仍有 16px 空间
```

### 6.6 卡片上下内边距专项规则

卡片内部不只检查左右边距，还必须检查上下边距。尤其是保修信息、附件摘要、字段表、更多入口等信息密集区域，最后一行内容不能贴近卡片底部。

强制规则：

```text
卡片标题距离卡片顶部：至少 20px，推荐 22–24px
标题与首个内容块间距：至少 14px，推荐 16px
二级状态条距离上方标题：至少 12px，推荐 14–16px
状态条内部上下内边距：至少 10px，推荐 12px
状态条与后续字段行间距：至少 14px，推荐 16px
字段行之间垂直间距：至少 6px，推荐 8–10px
最后一行内容距离卡片底部：至少 16px
摘要文案距离卡片底部：至少 16px
卡片与下一张卡片之间：至少 16px，推荐 18–22px
卡片与 composer 之间：至少 12px，推荐 16px
```

禁止：

- 字段行贴近卡片底部。
- 摘要文案压在卡片下边缘。
- 为了容纳更多内容而牺牲上下留白。
- 卡片内部只有左右 padding，没有上下 padding。
- 标题、状态条、字段行之间距离过近，形成一坨信息。

验收标准：

```text
1. 卡片顶部、标题、内容块、底部都有明确留白
2. 最后一行文字下方仍有 16px 左右空间
3. 状态条内部上下不贴边
4. 字段组不贴状态条，也不贴卡片底部
5. 相邻卡片之间有稳定垂直节奏
```

## 7. 按钮规则

### 7.1 默认页最多一个主按钮

主按钮样式：

```text
background: #111111
text: #FFFFFF
height: 40–44px
radius: 12–14px
font-size: 15px
font-weight: Semi Bold
```

规则：

- 默认页最多一个黑色主按钮。
- 主按钮应对应当前屏幕主任务。
- 不要把多个黑色按钮放在一屏。

### 7.2 中文长按钮不要横排

禁止在移动端横排两个长中文按钮，例如：

```text
[保存故障记录] [复制售后文案]
```

应改为：

```text
[保存故障记录]
[复制售后文案]
```

或默认页只保留主按钮，把次操作放入入口中。

### 7.3 次按钮和低强调按钮

次按钮：

```text
白底
描边 #E5E7EB
黑色文字
高度 40–44px
```

低强调按钮：

```text
无背景或透明 hit area
灰色文字 #6B7280
高度 30–36px
居中
```

示例：02 的「取消」必须是卡片内操作区的一部分，不得成为飘在卡片外的小字。

## 8. Icon 规则

### 8.1 所有 icon 必须使用 SVG / 矢量图形

禁止使用以下字符作为 icon：

```text
▣  □  ◇  ◌  ↑  !  ✓
```

规则：

- icon 尺寸通常为 20px 或 24px。
- icon 必须位于圆形 / 卡片 / 按钮中心。
- 不允许出现多余一撇、多余线段。
- 不允许出现蓝色选择框、蓝色描边残留。

### 8.2 图标验收标准

每个 icon 必须检查：

```text
1. 是否为 SVG 或基础矢量
2. 是否居中
3. 是否与文字基线对齐
4. 是否没有多余线段
5. 是否没有字符占位感
```

## 9. 警示 / 风险提示规则

### 9.1 风险提示要克制

风险提示使用低饱和语义色，不做大面积刺眼红。

推荐：

```text
background: #FEF2F2
border: #FECACA
text: #991B1B
radius: 12px
```

### 9.2 警示卡排版

推荐结构：

```text
[warning icon]  安全提醒
                先停止使用并关闭进水阀，
                再联系官方售后。
```

规则：

- 标题、正文、icon 不能挤在一起。
- 正文最多 2 行。
- 文案要短，不要把完整解释塞进警示卡。
- 卡片高度通常 56–72px。

---

## 10. Chips / 标签规则

### 10.1 默认页 chip 数量控制

一行 chip 不得过满。

规则：

```text
默认页一行最多 3 个 chip
chip 间距至少 8px
右侧不得贴边
```

如果超过 3 个：

- 删除低优先级 chip。
- 移到详情弹窗。
- 分成两行，但要保证底部不拥挤。

### 10.2 Chip 样式

```text
height: 22–24px
radius: 11–12px
background: #F4F4F2
font-size: 12px
text: #6B7280
```

## 11. 详情态 / Sheet 规则

### 11.1 什么时候必须使用 Sheet

以下内容必须进入 Sheet 或详情页：

- 全部设备字段。
- 完整售后沟通文案。
- 完整材料清单。
- 执行过程日志。
- 识别置信度详情。

### 11.2 Sheet 基本结构

```text
半透明遮罩
底部弹窗
grabber
标题
关闭按钮
正文 / 字段列表
主按钮
```

推荐：

```text
sheet height: 520–560px
radius top: 26px
左右边距: 28px
主按钮贴近底部但保留安全边距
```

---

### 11.3 上传 Action Sheet 规则

底部 Composer 左侧 `+` 点击后必须打开上传 Action Sheet。该 Sheet 是自动建档主链路入口，不得省略。

必须包含：

```text
标题：添加资料
拍照
从相册选择
上传文件
手动添加设备
取消
```

规则：

- Sheet 从底部出现，顶部有 grabber。
- 每个操作项高度 48–52px，适合拇指点击。
- 图标必须使用 SVG / 矢量，不使用字符占位。
- 「取消」使用低强调样式，放在操作列表底部。
- 不得把拍照、相册、文件上传拆散到多个不相关入口。

### 11.4 UploadedFileCard 状态规则

上传文件必须有明确状态，不允许只展示文件名。

必须覆盖：

```text
待上传
上传中
已上传
解析中
已解析
解析失败
```

每张文件卡至少包含：

```text
文件缩略图 / 文件类型图标
文件名
文件类型或大小
当前状态
删除 / 重试入口（按状态展示）
```

解析失败时必须提供恢复入口：

```text
重新上传
手动填写
仅保存附件
```

规则：

- 文件卡片不得挤压 Composer。
- 多文件时纵向排列，必要时折叠到 Sheet 或列表。
- `仅保存附件` 属于写入行为，必须进入确认流程。

### 11.5 执行过程 Sheet 规则

执行过程默认不展开，只在状态行或结果卡中提供轻量入口。

入口推荐：

```text
[✓] 已完成分析              [执行过程 >]
```

Sheet 必须展示：

```text
任务标题
当前状态
节点列表
节点状态
错误摘要（如有）
关闭按钮
```

节点状态包括：

```text
已完成
执行中
等待确认
失败
跳过
```

规则：

- 执行过程是可解释性辅助，不应抢占默认页主任务。
- 节点列表使用轻量时间线或列表，不使用复杂后台表格。
- 失败节点必须展示简短原因和下一步恢复建议。

## 12. 页面级规则

### 12.1 01 Agent Home / Idle

必须保持：

- 首页主问题清晰。
- 快捷建议 icon 居中、风格一致。
- 不出现标题蓝线。
- 关注卡和最近任务卡不被 composer 遮挡。

### 12.2 02 Agent / Create Device Confirmation

默认页只展示设备草稿摘要。

必须包含：

```text
设备草稿
设备名称
设备摘要
保修状态
已识别附件
建议提醒
缺失字段

```

规则：

- 不默认展开全部字段。
- 「取消」必须是卡片内低强调按钮，不是卡片外飘字。
- 操作区应是一个整体。
- 详细字段进入 02B Sheet。

### 12.3 03 Agent / Troubleshooting Result

默认页目标：让用户知道现在怎么处理故障。

必须优先展示：

```text
当前设备
用户问题
已完成分析
风险提示
先做这几件事
售后文案入口
需要准备摘要
保存故障记录
```

规则：

- 行动清单是视觉核心。
- 默认页只保留一个主按钮：保存故障记录。
- 复制售后文案通过「售后沟通文案」入口完成。
- 不横排两个长按钮。
- 完整售后文案进入 03B Sheet。

### 12.4 00 App Shell / SideDrawer

左侧抽屉是 v0.1 的主导航结构，必须补齐设计稿，不得用底部 Tab 替代。

必须至少包含 3 个画板：

```text
00 App Shell / Drawer Closed
00B App Shell / Drawer Open
00C Drawer Item Selected
```

抽屉结构：

```text
HomeCare Agent

新任务

主要
- 设备库
- 提醒
- Agent 执行记录

管理
- 设置
- 关于 HomeCare Agent
```

推荐规格：

```text
抽屉宽度：300–320px
背景：#FFFFFF
遮罩：rgba(0, 0, 0, 0.18)
菜单项高度：44–48px
左右内边距：24px
分组标题字号：12px
菜单文字字号：15px
当前项背景：#F4F4F2
当前项圆角：12px
```

交互规则：

- 点击左上角菜单打开抽屉。
- 点击遮罩关闭抽屉。
- 点击「新任务」回到 Agent 主界面并清空当前输入。
- 点击设备库、提醒、执行记录、设置、关于 HomeCare Agent 后关闭抽屉并进入对应页面。
- 当前页面对应菜单项需要轻量高亮。
- 数据导出不作为左侧抽屉独立入口，统一放入设置页。

### 12.5 Device Selection / 设备选择态

当用户问题无法唯一匹配设备时，必须先让用户选择设备，不得由 Agent 猜测。

适用场景：

```text
manual_qa
warranty_check
troubleshooting
```

推荐结构：

```text
你说的是哪台设备？

候选设备卡片列表：
- 设备名称
- 品牌 / 型号 / 分类
- 保修状态

操作：
- 选择设备
- 都不是
- 新建设备
```

规则：

- 候选设备默认最多展示 3 台，更多进入 Sheet。
- 每个候选项整卡可点。
- 不要在用户选择前生成说明书答案、保修结论或故障建议。

### 12.6 Manual Answer / 说明书问答状态

说明书问答必须同时设计成功态和异常态。

必须包含：

```text
成功回答
无说明书
未检索到答案
来源详情 Sheet
设备不明确选择态
```

无说明书文案：

```text
这台设备还没有上传说明书。
上传说明书后，我可以帮你回答使用方法、故障代码和保养步骤。
```

未检索到答案文案：

```text
我没有在这台设备的说明书中找到明确答案。
你可以换个问法，或上传更完整的说明书。
```

规则：

- 成功回答必须展示来源入口。
- 不要用大段文本铺满默认页。
- 来源片段、页码、附件名进入 Sheet。

### 12.7 Save Maintenance Record Confirmation

```text
取消
```

规则：

- 默认页主按钮仍是「保存故障记录」。
- 点击后进入确认态，确认前不写入 MaintenanceRecord。
- 取消后返回故障结果，不丢失结果内容。

## 13. Figma 交付规则

### 13.1 不得留下编辑态残留

交付前必须：

```text
figma.currentPage.selection = []
```

并检查：

- 无蓝色选中框。
- 无蓝色描边。
- 无临时辅助线。
- 无未命名占位图层。


### 13.2 Icon 修复专项规则

所有状态、跳转、关闭、发送类 icon 必须使用矢量图层或可复用 icon 组件，不允许使用字符、旋转线段或临时占位图层模拟。

重点检查对象：

```text
状态 check icon
右侧 chevron icon
Sheet close icon
Composer send icon
Drawer menu icon
执行路径时间线 icon
```

交付前必须确认：

```text
1. check icon 不得显示成歪斜线段或绿色碎线。
2. chevron icon 必须是标准右箭头，不得像对勾或残缺折线。
3. close icon 必须是居中的 X，不得只剩单边斜线。
4. send icon 必须是居中的白色上箭头，不得偏移或像对勾。
5. 抽屉、设置页、执行记录列表、状态行、来源入口都必须使用同一套 icon 规范。
```

### 13.3 图层命名

图层命名必须可读。

推荐：

```text
status-time
status-wifi-icon
top-title
user-bubble
result-card
primary-action-bg
primary-action-text
composer-bg
composer-send-icon
```

禁止：

```text
Rectangle 123
Vector 45
文字 1
icon-□
```

### 13.3 修改后必须截图验收

每次修改 Figma 后都要导出当前画板截图检查：

```text
1. 是否有溢出
2. 是否有遮挡
3. 是否有蓝色残留
4. icon 是否正常
5. 文字是否自然换行
6. 按钮是否位于正确容器内
7. composer 是否遮挡内容
```

## 16. 强制设计原则

后续 HomeCare Agent 所有设计稿必须遵守以下原则：

```text
不要硬塞内容。
不要用字符当 icon。
不要让按钮掉出卡片。
不要让 composer 和内容打架。
不要横排长中文按钮。
不要让链接漂浮。
不要把默认页做成详情页。
不要留下蓝色编辑态。
```

更好的做法是：

```text
默认页轻量。
核心操作明确。
详情进入 Sheet。
组件复用已验收版本。
每次改完截图验收。
```
